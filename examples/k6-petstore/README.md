# k6-petstore example

End-to-end demonstration of the `@ahmedrowaihi/k6-*` track against the public petstore3 demo server.

## What's here

- [k6-tools.config.ts](k6-tools.config.ts) — points the CLI at `../../fixtures/petstore.yaml`.
- [src/loadtest.ts](src/loadtest.ts) — user-owned entry. Four named scenarios (`browse`, `write`, `stressTest`, `spikeTest`) with flat budgets, per-op overrides, and step chaining.
- `src/gen/` — generated by `k6-tools sync`. Re-runs whenever the spec changes; `loadtest.ts` stays untouched.

## Run it

```bash
# from repo root
pnpm install

cd examples/k6-petstore

# (Re-)generate the typed client from fixtures/petstore.yaml.
pnpm sync

# Smoke: 1 VU × 30s, drives lt.default which is the first scenario (browse).
pnpm run:smoke

# Other pace flavors — CLI overrides drive k6's implicit `default` scenario.
pnpm run:load
pnpm run:stress
pnpm run:spike
```

`pnpm run` (no flags) executes all four scenarios concurrently because k6 honors every entry in the `options.scenarios` block.

## What you're seeing

`src/loadtest.ts` is the *only* file the developer writes. The generated client (`src/gen/client.ts`) gives one typed function per spec operation; the rest is composition:

```ts
flow()
  .step("create",  () => addPet(data.Pet()))    // Pet
  .step("read",    (pet) => getPetById(pet.id!)) // pet typed end-to-end
  .step("update",  (pet) => updatePet({ ...pet, status: "sold" }))
  .expect((pet) => pet.status === "sold")
  .step("cleanup", (pet) => deletePet(pet.id!));
```

`pet.id` is `number` because the spec said so. If the spec renames `id`, the next `k6-tools sync` causes a TypeScript error pointing at this exact line.

The public petstore3 demo server is intermittently flaky on a handful of endpoints (`/store/inventory`, `/pet/findByStatus` periodically 500). Budgets in this example are loose to keep the demo runnable; tighten them against your own backend.
